# PDIS 个人决策小秘书 - 需求文档 V2

## 文档信息
- **版本**: V2.0 (基于代码实际能力修订)
- **日期**: 2026-04-27
- **状态**: 当前代码可实现
- **适用范围**: 单用户本地版

---

## 一、产品定义 (修订)

### 1.1 一句话描述
PDIS是一个基于本地大模型的**个人决策辅助工具**，帮助用户分析职场/人际场景中的决策问题，提供可行性判断和行动建议。

### 1.2 核心定位
- **目标用户**: 个人用户，单用户本地方案
- **使用场景**: 职场决策、人际沟通、重要选择
- **核心能力**: 基于人物性格分析给出决策建议
- **技术方案**: 本地Ollama模型，无需联网

### 1.3 与V1原规划的差异

| 原规划 | 当前实际 | 调整说明 |
|--------|----------|----------|
| 多用户体系 | 单用户 | V2只支持单用户本地使用 |
| Web界面优先 | CLI优先 | 先跑通核心流程，界面后续迭代 |
| PostgreSQL存储 | SQLite/JSON | 简化部署，单文件存储 |
| 向量数据库检索 | 关键词匹配 | MVP阶段暂不引入向量库 |
| 四LLM独立部署 | 2-3模型复用 | 根据实际模型能力调整分工 |

## 二、用户需求场景

### 2.1 核心场景 (必须支持)

**场景1: 职场决策分析**
```
用户输入: "明天要和老板谈加薪，她最近因为我项目延期不太高兴，我该去吗？"

系统输出:
- 可行性判断: 有条件可行
- 风险提醒: 延期负面情绪+立即谈加薪=风险较高
- 建议策略: 先道歉复盘，再谈发展，暂缓直接提加薪
- 话术建议: [具体沟通话术]
```

**场景2: 人物档案建立**
```
用户输入: "我们CTO叫张伟，性格强势，结果导向，上次我PPT没做好被他批评了"

系统处理:
- 创建"张伟_CTO"人物档案
- 提取性格特征: 强势、结果导向
- 记录互动历史: PPT批评事件
- 生成性格标签: 领导型(leadership)
```

**场景3: 历史决策参考**
```
用户输入: "上次你说我那个加薪谈判方案，我想再看看"

系统处理:
- 检索历史决策记录
- 展示之前的分析和建议
- 询问: "情况有变化吗？需要重新分析吗？"
```

### 2.2 支持场景 (尽力支持)

- 多轮对话澄清信息
- 人物关系积累(越用越准)
- 相似场景建议复用

### 2.3 不支持场景 (明确拒绝)

- 多人协作共享
- 云端同步/备份
- 手机APP推送
- 实时数据查询(天气/股价等)

### 3.2 功能清单

#### 3.2.1 决策分析功能 (P0 - 必须有)

| 功能 | 需求描述 | 验收标准 |
|------|----------|----------|
| 自然语言输入 | 用户用自然语言描述决策场景 | 支持中文自由文本输入 |
| 人物识别 | 自动提取场景中涉及的人物 | 能识别人物称谓、角色 |
| 场景理解 | 理解决策的核心问题 | 能提取时间、地点、事件 |
| 信息追问 | 信息不足时主动提问 | 能问出关键缺失信息 |
| 可行性分析 | 给出决策可行性判断 | 输出:可行/有条件可行/不可行 |
| 行动建议 | 给出具体执行步骤 | 步骤<=5条，可执行 |
| 风险提示 | 列出主要风险点 | 风险>=1条，具体明确 |
| 结果保存 | 保存分析结果供后续查看 | 可检索历史分析 |

#### 3.2.2 人物档案功能 (P0 - 必须有)

| 功能 | 需求描述 | 验收标准 |
|------|----------|----------|
| 人物创建 | 新建人物档案 | 包含:姓名/角色/关系 |
| 性格分析 | 基于描述生成性格标签 | 使用九型人格分类 |
| 互动记录 | 记录与该人物的重要事件 | 时间+事件+结果 |
| 档案查询 | 查看已建立的人物档案 | 列表+详情展示 |
| 档案更新 | 补充新信息到现有人物 | 版本追加，不覆盖 |

#### 3.2.3 对话交互功能 (P1 - 应该有)

| 功能 | 需求描述 | 验收标准 |
|------|----------|----------|
| 多轮对话 | 支持连续对话澄清 | 上下文连贯 |
| 意图识别 | 识别用户是闲聊/决策/查档案 | 正确分类 |
| 会话保存 | 保存对话历史 | 可查看完整对话 |
| 流式输出 | 逐步显示思考过程 | 非等待全部完成 |

#### 3.2.4 记忆检索功能 (P1 - 应该有)

| 功能 | 需求描述 | 验收标准 |
|------|----------|----------|
| 相似场景匹配 | 检索历史相似决策 | 相关性排序 |
| 人物自动召回 | 自动关联相关人物档案 | 基于关键词匹配 |
| 上下文压缩 | 长对话自动摘要 | 保留关键信息 |

### 3.3 明确不支持的功能 (Out of Scope)

| 功能 | 说明 | 原因 |
|------|------|------|
| 用户注册登录 | 无账号体系 | 单用户本地版 |
| 多设备同步 | 无云端存储 | 数据仅本地 |
| 权限管理 | 无角色区分 | 单用户无需 |
| 数据导出PDF | 仅支持JSON导出 | 简化优先级 |
| 定时提醒 | 无推送能力 | 纯对话交互 |
| 语音输入 | 仅文本输入 | 简化优先级 |

## 五、数据需求

### 5.1 核心数据实体

```
人物档案(PersonProfile)
├── person_key: string          # 唯一标识
├── name: string                # 显示名称
├── role: string                # 角色(老板/同事等)
├── relationship: string        # 与用户关系
├── personality: {
│   ├── color: string          # 九型人格类型
│   ├── original: string       # 中文术语
│   ├── strengths: [string]   # 优势
│   └── weaknesses: [string]   # 劣势
├── interactions: [{
│   ├── date: string           # 时间
│   ├── event: string          # 事件
│   └── outcome: string        # 结果
}] └── created_at: string       # 创建时间
├── updated_at: string          # 更新时间

决策记录(DecisionRecord)
├── id: string                  # 唯一ID
├── user_input: string          # 用户原始输入
├── scenario: string            # 场景摘要
├── persons_involved: [string] # 涉及人物
├── feasibility: string         # 可行性结论
├── confidence: float          # 信心度
├── key_reasons: [string]      # 关键原因
├── execution_plan: [string]    # 执行计划
├── risks: [string]            # 风险点
├── created_at: string         # 创建时间
└── follow_up: string          # 后续跟进状态
```

### 5.2 存储方案

```
当前实现: JSON文件
├── person_registry.json        # 人物档案库
├── decision_history.json       # 决策历史
└── full_reports.json           # 完整报告(现有)

未来演进: SQLite
├── 单文件数据库
├── 支持简单查询
└── 无需额外服务
```

## 六、交互流程

### 6.1 标准决策流程

```
用户: 我想明天和老板谈加薪

系统: [LLM-A识别: 人物=老板, 决策=谈加薪, 时间=明天]
      [检索人物档案: 未找到老板档案]
      
      为了给你更准确的建议，我需要先了解你老板的一些信息：
      1. 老板是男是女？性格大概怎样？(比如强势、温和、理性？)
      2. 你们平时的沟通方式是怎样的？
      
用户: 老板是女的，比较强势，对结果很看重

系统: [LLM-C创建老板档案: 性格=领导型]
      [LLM-B检查: 仍缺少信息-加薪理由、入职时长等]
      
      了解了。另外几个问题：
      1. 你入职多久了？
      2. 有加薪的硬性理由吗？(比如满年限/有offer/业绩突出)
      
用户: 入职2年，有外部offer，薪资高30%

系统: [LLM-D决策分析]
      
      【决策分析结果】
      可行性: 中高 (75%)
      
      关键判断:
      - 两年周期是合理调薪节点 ✓
      - 外部offer提供谈判筹码 ✓
      - 老板强势风格需要策略应对 ⚠
      
      执行建议:
      1. 将谈话定位为"职业发展讨论"
      2. 委婉提及外部机会，表达"优先考虑内部"
      3. 给老板选择空间：匹配或制定调薪计划
      
      风险提示:
      - 强势型老板可能视为"威胁"
      - 建议准备Plan B(是否接受外部offer)
      
      需要我帮你准备具体话术吗？
```

### 6.2 人物建档流程

```
用户: 记录一下我们CTO

系统: 好的，请描述一下这位CTO：
      - 姓名或称谓
      - 性格特点
      - 你们的互动经历
      
用户: 叫张伟，性格强势，结果导向，上次我PPT没做好被他批评了

系统: [LLM-C分析: 领导型]
      
      已创建张伟_CTO的人物档案：
      - 性格: 领导型(强势、结果导向)
      - 互动记录: PPT批评事件
      
      以后涉及和张伟的决策，我会参考这个档案给出建议。
```

### 6.3 历史查询流程

```
用户: 查看我之前关于加薪的分析

系统: [检索历史记录]
      找到3条相关记录：
      1. 2026-03-15 与老板谈加薪分析
      2. 2026-02-20 跳槽vs留任决策
      3. 2026-01-10 年终调薪建议
      
      想看哪一条？或者告诉我更具体的关键词。
      
用户: 第1条

系统: [展示完整分析结果]
      这是2026-03-15的分析...
      
      当时的情况是：...
      建议是：...
      
      已经过去6周了，情况有变化吗？需要重新分析吗？
```

## 八、风险与限制

### 8.1 已知限制

1. **模型能力依赖**: 分析质量取决于本地模型能力
2. **无实时数据**: 无法获取天气、股价等实时信息
3. **单用户设计**: 不支持多人共享或协作
4. **无网络能力**: 无法搜索互联网信息

### 8.2 使用边界

- 仅提供决策参考，不构成专业建议
- 用户需自行判断最终决策
- 敏感决策(医疗/法律)建议咨询专业人士
