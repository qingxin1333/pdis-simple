# Personal Decision Intelligence System（PDIS）

## 概要设计文档（High-Level Design）

**文档级别**：概要设计
 **目标读者**：架构师、开发组长、技术负责人
 **版本**：V1.0
 **范围**：单用户、本地优先、多 LLM 协同的决策分析系统

------

## 1. 设计目标与背景

### 1.1 背景说明

在现实决策场景中（人际沟通、关键选择、复杂判断），用户往往难以进行系统性分析。传统问答式 AI 或规则系统无法满足以下需求：

- 复杂语境下的角色关系识别
- 基于历史人物信息的连续性分析
- 信息不完整情况下的主动追问
- 从分析到**可执行行动方案**的输出

PDIS 的设计目标，是构建一套**以人工智能为主导分析主体**的系统，使 AI 能够在结构化上下文支持下，完成接近“人类专家级”的决策推演。

------

### 1.2 设计目标

PDIS V1 的概要设计目标包括：

1. 构建一个**人机协作的决策分析平台**，而非简单对话系统
2. 通过多 LLM 分工协作，提高分析稳定性、可控性与可扩展性
3. 将用户的自然语言输入转化为可供 AI 分析的结构化上下文
4. 在信息不足时，由 AI 主动驱动补充交互
5. 输出结构化、可执行的决策分析结果
6. 为后续多用户、隐私、审计等能力预留架构扩展空间

------

## 2. 系统总体架构概览

### 2.1 架构风格

PDIS 采用 **分层架构 + 编排式 AI 调度（Orchestrated AI Pipeline）** 的设计风格。

系统不是单一 LLM 服务，而是一个 **LLM 矩阵**，由后端统一编排、调度和状态管理。

------

### 2.2 高层架构视图（逻辑）

```
┌──────────────────────────┐
│        Client Layer       │

             │

```

## 3. 核心设计思想

### 3.1 “AI 为分析主体”的系统定位

在 PDIS 中：

- AI 不是被动工具
- AI 是分析与判断的主要执行者
- 系统的职责是为 AI 提供 **正确、完整、结构化的分析上下文**

因此，系统设计的重点不在于 UI 或规则逻辑，而在于：

- 上下文构建能力
- 分析条件判断能力
- AI 调度与流程控制能力

------

### 3.2 多 LLM 协同而非单模型

PDIS 明确避免“一个模型解决所有问题”的设计。

原因包括：

- 不同任务对模型能力侧重不同
- 小模型在结构化抽取上更稳定
- 大模型在复杂推理上更具优势
- 分工有利于性能、成本与稳定性控制

------

## 4. 多 LLM 协同设计（核心）

### 4.1 LLM 职责划分

| 模型      | 定位             | 核心职责                                             |
| --------- | ---------------- | ---------------------------------------------------- |
| **LLM A** | 输入解析模型     | 拆解用户输入，抽取人物、时间、地点、场景、角色等要素 |
| **LLM B** | 分析条件判断模型 | 判断是否满足决策分析条件，生成补充问题               |
| **LLM C** | 人物分析模型     | 生成人物性格色彩报告与人物侧写报告                   |
| **LLM D** | 决策分析模型     | 输出重大决策的可行性分析与执行建议                   |

------

### 4.2 LLM 协同原则

- **单一职责**：每个 LLM 只承担一种明确职责
- **结构化输入输出**：LLM 之间通过 JSON 进行交互
- **后端统一编排**：LLM 不直接相互调用
- **可替换性**：模型可按能力与成本替换，不影响整体流程

------

## 5. 核心业务流程（概要）

### 5.1 决策分析主流程（概览）

```
User Input
   ↓
LLM A: Input Decomposition
   ↓
Profile Matching & Context Assembly
   ↓
LLM B: Analysis Readiness Check
   ↓
[Optional Loop: AI Question → User Answer]
   ↓
LLM C: Profile Analysis (if needed)
   ↓
LLM D: Decision Analysis
   ↓
Persist & Respond
```

------

### 5.2 流程设计要点

- **循环可控**：补充问答有最大轮次限制
- **条件驱动**：是否进入下一阶段由 LLM B 决定
- **状态可追踪**：每次分析均有明确阶段状态
- **中断友好**：用户可在任意补充阶段终止流程

------

## 6. 数据与上下文设计（概要）

### 6.1 核心数据对象

- **Person Profile**：人物基础信息
- **Personality Report**：性格色彩报告（结构化）
- **Profiling Report**：人物侧写报告（结构化）
- **Decision Case**：一次完整的决策分析实例
- **Analysis Context Snapshot**：分析时的上下文快照

------

### 6.2 不可变性原则（V1）

- 历史报告不可被覆盖
- 决策分析结果只读
- 新信息通过新版本追加

该原则有利于：

- 分析可回溯
- 后续审计扩展
- AI 行为可解释

------

## 7. 系统边界与非目标

### 7.1 V1 明确不承担的职责

- 权限与隐私治理
- 多用户数据隔离
- 审计合规展示
- 人工干预决策结果

------

## 8. 可扩展性与演进设计

### 8.1 架构预留点

- LLM 调度层可扩展为策略引擎
- 上下文构建层可引入向量检索
- 分析记录可用于后续效果评估
- 用户反馈可作为模型微调数据源

------

## 9. 关键风险与控制点（概要）

| 风险             | 应对设计                    |
| ---------------- | --------------------------- |
| LLM 输出不稳定   | JSON 强约束 + Repair Loop   |
| 信息不足导致误判 | LLM B 条件判断 + 主动追问   |
| 模型性能差异     | 多模型分工与可替换          |
| 上下文污染       | 不可变快照 + 最小上下文原则 |

------

## 10. 总结

PDIS 并非传统意义上的 AI 应用，而是一套 **以人工智能为分析主体、以系统为协作与编排平台** 的决策支持架构。

通过多 LLM 协同、结构化上下文驱动与条件控制流程，PDIS 在 V1 阶段即可实现对复杂现实决策问题的系统性分析，为后续能力扩展奠定稳定架构基础。****
