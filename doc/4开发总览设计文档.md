  # 一、关键技术清单（版本建议，供 V1 使用）

  - **向量 DB**：Milvus **2.6.x**（推荐 2.6+）
  - **LLM 推理管理**：Ollama **0.15.6**（用于本地模型加载和 API 调度）
  - **模型（V1 建议组合，用于不同业务职责）**：
    - Qwen2.5-7B（7.6B，适合抽取/轻量推理）
    - Qwen3-8B（8.2B，主分析模型）
    - ChatGLM-6B（6.2B，人物侧写/对话任务可量化运行）
  - **前端环境**：Node.js **22.x（22.22.0 建议）**
  - **ORM 与迁移**：SQLAlchemy 2.0+ + Alembic（与 PostgreSQL 配合）
  - **向量扩展**：pgvector（可选，用于 Postgres 内向量检索，V1 建议 Milvus 专用）

  ## **WSL2 Ubuntu 24.04**

  1. **技术栈的 Linux 依赖性**：
   - 从您提供的文档中可以看出，PDIS 的技术栈（PostgreSQL 15.x、Milvus 2.6.x、Ollama）在 Linux 环境下有最佳支持
     - 您的文档中明确建议使用 "PostgreSQL 15.x"、"Milvus 2.6.x"，这些在 Linux 上的安装和配置比 Windows 更简单、更稳定
   - 特别是 Milvus（向量数据库）在 Linux 上有官方支持，Windows 支持有限
  2. **Docker 集成问题**：
   - 您的 Docker Desktop 已经在 Windows 上运行，但 Docker 容器本身是 Linux 容器
     - WSL2 提供了更好的 Linux 容器支持，避免了 Windows 和 Linux 容器之间的兼容性问题
     - 文档中提到的 "Docker 29.x + Docker Compose 5.x" 在 WSL2 中运行更流畅
  3. **Python 环境一致性**：
     - 您的文档明确要求 "Python 3.12.x + FastAPI 0.111.x"
     - 在 Windows 上安装某些 Python 包（特别是涉及 C 扩展的包）可能会遇到问题
   - WSL2 提供了与生产环境一致的 Linux 环境，避免了"在我机器上可以运行"的问题
  4. **Ollama 和模型部署**：
   - 您的文档建议使用 "Qwen2.5-7B"、"Qwen3-8B" 等模型
     - Ollama 在 Linux 上对这些模型的支持更完善，特别是对于本地模型部署
   - 文档中提到的 "Ollama 0.15.6" 在 WSL2 中运行更稳定

  # 二、业务逻辑（Business Logic）—— 用一句话概括
  
  **PDIS 的业务目标**是将用户以自然语言提出的“现实决策问题”转换为“结构化上下文 + 相关人物档案”，由多模型协同完成“条件判断 → 深度分析 → 可执行建议”的闭环输出；在信息不足时，系统自动生成针对性的补充问题并支持多轮交互，直到分析条件满足或用户终止。
  
  ### 业务规则（关键点）
  
  1. **单次事务以一次决策请求为单位**（request-id），整个分析过程可被完整记录并只读保存为历史记录。
  2. **上下文版本化**：人物档案与侧写报告都是只读快照（版本化）——同一人物在不同时间可能有多个版本供历史分析参考。
  3. **最少用户干预**：系统优先尝试由 LLM 完成判断/补全，仅在明确缺少“可回答的事实字段”时才向用户发问。
  4. **可审计的推理流**：每一步 LLM 输入（prompt）与输出（response）均记录，便于事后复盘与模型调优。
  5. **安全边界**：V1 阶段默认单用户、私有部署；敏感数据原则上不出本地（若混用云端 API，需脱敏/加密）。

  ## **四、流程描述示例（完整修正版）**
  
  ### **📥** ***\*用户输入\****
  
  > “张总昨天在会上说，如果 Q2 交付再延迟，就要换掉项目经理。”

  ------

  ### **Step 1: LLM A（输入结构化）**
  
  - **输入**：`"张总昨天在会上说，如果 Q2 交付再延迟，就要换掉项目经理。"`
  
  - 输出
  
    ```
    {
      "identified_persons": ["张总"],
      "time_info": "2026-04-30",
      "scenario_summary": "张总威胁因Q2交付延迟而更换项目经理"
    }
    ```

  - **数据库交互**：**无**（仅生成文本，不写入任何库）

  - **关键操作**：系统用 `scenario_summary` 生成查询向量（如 `embed("张总威胁因Q2交付延迟...")`）
  
  ------
  
  ### **Step 2: 人物匹配引擎（Milvus + PostgreSQL）**
  
  - Milvus 检索
  
    - 查询向量：`embed("张总威胁因Q2交付延迟...")`
  
    - Milvus 中已有数据（由 LLM C 之前写入）：
  
      表格
  
      | person_id | vector      | text_source                     |
      | :-------- | :---------- | :------------------------------ |
      | 101       | [0.23, ...] | "张伟，CTO，性格强势，关注交付" |
    | 102       | [0.88, ...] | "李娜，HRD，温和"               |
  
  - **Milvus 返回**：`person_id=101`（相似度 0.92）
  
  - PostgreSQL 查询
  
    ```
    SELECT * FROM person_profile WHERE id = 101 AND deleted = false;
    ```
  
    返回
  
  ```
    {
    "id": 101,
      "name": "张伟",
      "title": "CTO",
      "profile_json": {
        "personality_color": "红色（支配型）",
        "risk_tolerance": "低"
      },
    "version": 3
    }
  ```
  
  ------
  
### **Step 3: LLM B（判断是否 ready）**
  
- **输入**：LLM A 输出 + 张伟档案
  
- **发现缺失**：缺少“当前项目经理”信息
  
- 输出
  
    ```
    {
      "ready": false,
      "followup_questions": ["请问当前的项目经理是谁？"]
    }
    ```
  
  ------
  
  ### **Step 4: 用户补充**
  
  > “项目经理是王磊。”
  
  ------
  
  ### **Step 5: LLM C（人物侧写补全）**
  
  - **输入**：上下文 + “王磊是项目经理”
  
  - 输出新档案
  
    ```
    {
      "name": "王磊",
      "role": "项目经理",
      "inferred_traits": "抗压能力弱（被威胁替换）"
    }
    ```
  
  - 数据库写入
  
    1. PostgreSQL
  
       ```
       INSERT INTO person_profile (name, profile_json, version) 
       VALUES ('王磊', '{"role":"项目经理", "inferred_traits":"抗压能力弱"}', 1);
       -- 返回 person_id=205
       ```
  
    2. Milvus
  
       - 文本：`"王磊，项目经理，抗压能力弱"`
       - 向量化 → 写入 Milvus（关联 `person_id=205`）
  
  ------
  
  ### **Step 6: LLM D（决策分析）**
  
  - **输入**：完整上下文（含张伟档案 + 王磊档案）
  
  - 输出
  
    ```
    {
      "feasibility": "低",
      "confidence": 0.85,
      "execution_plan": "建议立即召开三方会议",
      "risks": ["张伟可能坚持换人", "王磊士气受挫"]
    }
    ```
  
  - 数据库写入
  
    1. PostgreSQL
  
       ```
       INSERT INTO analysis_result (input_text, result_json, model_used)
     VALUES ('张总昨天在会上说...', '{"feasibility":"低", ...}', 'llama3.1-8b');
       ```

    2. PostgreSQL

       ```
       INSERT INTO audit_log (action, entity_id, payload_hash)
       VALUES ('llm_analysis', 999, 'sha256(...)');
       ```

  ------

  ### **Step 7: 前端展示**

  - 输出

    > **可行性**：低
    > **建议话术**：“张总，我们理解您对交付的担忧，王磊已在调整计划...”
    > **风险点**：张伟性格强势（红色支配型），可能坚持换人
  
  - **历史记录**：可通过 `analysis_id` 查询 `analysis_result` 表（只读）
  
  ### 每个节点的“成功/失败”判断
  
  - **抽取器成功**：识别到至少 1 个人物或场景关键要素；若否，触发补问（LLM B 可合并此职责）。
  - **人物匹配成功**：从人物库找到候选并返回匹配置信度（>=阈值）。若低于阈值，提示用户确认或建档。
  - **分析准备就绪**：LLM B 返回 `ready: true`，否则返回一组明确、可回答的问题（每个问题附带字段类型与示例答案）。
  - **最终分析有效性**：LLM D 输出包含 `feasibility` + `confidence`（数值）+ `execution_plan`（步骤化），并且 `confidence >= minimal_threshold`（可配置，V1 建议 0.6）。

  # 四、用业务流程连接技术：逐步映射（Responsibilities → Tech）
  
  下面按**业务步骤**逐步列出所需技术组件与交互要点。
  
## 1) 用户输入 & 前端交互
  
- **职责**：收集原始文本、展示补问、展示最终分析结果与历史记录。
  - **前端技术**：Vue.js（Vite + Pinia 状态管理） + Element Plus（组件库）。
- **API 交互**：调用后端 REST 接口（/api/v1/decisions、/api/v1/profiles 等）。
  - **注意点**：前端需把用户输入做最小预校验（长度、必要字段标记），并显示进度（LLM 请求可能有延迟）。

  ## 2) 输入结构化（LLM A）

  - **职责**：实体抽取、共指消解、时间解析、场景要素化。
- **实现**：后端通过 Ollama API 将 prompt 转发到 `qwen2.5-7b-instruct`（或云端轻量模型）并解析返回 JSON。
  - **数据持久化**：临时存储在 FastAPI 的会话上下文或 Redis（V1 若无高并发可先仅内存/DB）。
  - **技术要点**：建议把抽取得到的向量化表示同时写入 Milvus（用于后续相似性检索 / candidate matching）。
  
## 3) 候选人物匹配（Postgres + Milvus）
  
- **职责**：基于抽取出的 `mention` 与特征在人物库中匹配历史档案。
  - **实现**：
    - Postgres 存储结构化元数据（name, relation, version, metadata）；
  - Milvus 存储人物向量（embedding），提供相似度检索；后端将 top-N 候选返回给 Orchestrator。
  - **技术点**：同步策略（写入 Postgres 后触发异步任务将文本 embedding 写入 Milvus）。

  ## 4) 分析条件判断与追问（LLM B）

  - **职责**：判断是否进入最终分析；若不足，生成可立即展示给用户的补问列表。
  - **实现**：调用 `qwen3-8b`（或混用云端 GPT-3.5）来执行高可靠性条件判断，输出 `ready` + `missing_fields` + `followup_questions` JSON。
  - **交互**：若需要追问，前端按问题逐条收集并回传；Orchestrator 将新的信息追加到上下文并重试 LLM A/B。
  
  ## 5) 人物侧写补全（LLM C）

  - **职责**：基于人物输入或新增事件更新人物侧写报告（版本化），并生成结构化分析字段（性格色彩、驱动因素等）。
- **实现**：`ChatGLM-6B`（量化后）或 `qwen2.5` 用于快速侧写；结果写入 Postgres（只读快照）并将 embedding 写入 Milvus。
  - **版本化**：每次更新生成新版本 ID，旧版本保留作为历史上下文。

  ## 6) 决策分析与执行建议（LLM D）
  
  - **职责**：基于最终结构化上下文产出 `feasibility`、`confidence`、`execution_plan` 与 `risks`。
  - **实现**：主力使用 `qwen3-8b`（可通过 Ollama 本地部署）执行深度推理；必要时并行调用云端高容量模型做验证/对比（可选）。
  - **输出格式**：统一 JSON Schema（便于前端渲染与审计存档）。同时记录推理用 prompt、model name、时间戳、token 消耗（如云端调用）到审计日志。
  
  ## 7) 结果存储与回放
  
  - **职责**：保存完整分析会话（输入、每次 LLM 输出、最终结论）作为只读历史。
  - **实现**：Postgres 存储元数据与 JSON Blob；若需要全文搜索或分析，可把文本索引到 Elasticsearch（后续扩展）。Milvus 存储 embedding、便于相似场景召回。
  - **合规**：对敏感字段做加密/脱敏（在写库前），使用访问控制（V1 简化版仅单用户环境可略）。

  # 五、数据架构与领域模型

  ## 5.1 领域模型核心实体

  根据业务需求，系统核心实体包括：**人物档案（Person）**、**事件记录（Event）**、**分析结果（Analysis）**、**审计日志（Audit）**、**系统配置（Preferences）**。实体关系如下：

  - 一个 Person 可关联多个 Event（人物相关事件）
  - 一个 Analysis 可引用多个 Person（匹配到的人物快照）和多个 Event（检索到的事件）
  - Audit 记录所有重要操作，不可变
  - Preferences 为单行配置表
  
  ## 5.2 实体详细定义
  
  ### 5.2.1 Person（person_profile）
  
  - **用途**：与用户有关的“人”的长期档案。每个人用不变的 `id` 唯一标识；其它属性可版本化更新。
- **关键属性**：
    - `id`（UUID，主键）
  - `name`（文本，当前显示名）
    - `aliases`（JSONB，别名数组）
  - `profile_json`（JSONB，侧写结构化数据：性格标签、常见反应等）
    - `tags`（文本数组）
    - `sensitivity`（smallint，敏感级别，预留）
    - `deleted`（smallint，软删除标志：0-正常，1-删除）
    - `created_time`、`updated_time`（带时区时间戳）
  
  ### 5.2.2 Event（event_record）
  
  - **用途**：原始事实、对话、邮件片段等“发生过的事情”，一经记录原文不可覆盖。
  - **关键属性**：
    - `id`（UUID）
  - `title`（文本，可选）
    - `event_date`（时间戳，事件发生时间）
  - `text`（文本，原始内容，不可覆盖）
    - `person_id`（UUID，关联人物，可为空）
  - `metadata`（JSONB，附加信息：来源、作者、标签等）
    - `embedding_id`（文本，向量库中的ID，或直接用 embedding 字段）
  - `embedding`（vector(1536)，若使用 pgvector）
    - `sensitivity`（smallint）
    - `deleted`（smallint）
    - `created_time`、`updated_time`
  
  ### 5.2.3 Analysis（analysis_result）
  
  - **用途**：一次用户输入触发的完整分析记录（输入、检索上下文、模型输出等），append-only 核心资产。
  - **关键属性**：
    - `id`（UUID）
    - `scenario_text`（文本，用户原始输入）
    - `context_overrides`（JSONB，用户补充信息）
    - `matched_profiles`（JSONB，匹配的人物快照ID列表）
    - `retrieved_event_ids`（JSONB，检索到的事件ID列表）
  - `prompt_summary`（文本，提示词摘要或模板ID）
    - `json_output`（JSONB，结构化输出）
  - `text_output`（文本，人类可读摘要）
    - `model_meta`（JSONB，模型名称、版本、本地/云标识）
  - `confidence_score`（float，置信度）
    - `repair_attempts`（int，重试次数）
  - `used_cloud`（smallint，是否使用云模型）
    - `status`（文本，processing/done/failed/requires_human_review）
    - `deleted`（smallint）
    - `created_time`、`updated_time`
  
  ### 5.2.4 Audit（audit_log）
  
  - **用途**：只读审计流水，记录重要操作（analysis 创建、cloud consent、repair 尝试、delete/restore 等）。
  - **关键属性**：
    - `id`（UUID）
    - `action_type`（文本，如 analysis.create / profile.update）
    - `entity_type`（文本，person/event/analysis/preferences）
    - `entity_id`（UUID，相关实体ID）
    - `payload_hash`（文本，变更哈希）
    - `meta`（JSONB，任意结构化数据）
    - `created_time`（时间戳）
  
  ### 5.2.5 System Preferences（system_preferences）
  
- **用途**：本地系统配置（单用户），如是否允许云端调用、首选本地模型等。
  - **关键属性**：
  - `id`（UUID）
    - `key`（文本，唯一）
  - `value`（JSONB，配置值）
    - `updated_time`（时间戳）

  ## 5.3 生命周期与状态迁移规则
  
  - **软删除**：所有业务表使用 `deleted`（smallint，0/1）标识删除，查询默认过滤 `deleted=0`。
  - **版本化**：人物侧写更新时，旧版本保留在历史表中（通过 `created_time` 区分）；Analysis 记录为一次快照，不更新。
  - **审计**：Audit 表仅追加，禁止更新/删除（通过数据库权限控制）。
  
  ## 5.4 数据库表设计（Postgres）
  
  采用 PostgreSQL 15+，启用 `pgcrypto` UUID 扩展，核心表结构如下（仅展示关键字段，完整 SQL 见附录）：

  sql

  ```
-- 人物档案表
  CREATE TABLE person_profile (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    name text NOT NULL,
  aliases jsonb DEFAULT '[]',
    profile_json jsonb DEFAULT '{}',
    tags text[] DEFAULT '{}',
    sensitivity smallint DEFAULT 0,
    deleted smallint DEFAULT 0,
    created_time timestamptz DEFAULT now(),
    updated_time timestamptz DEFAULT now()
  );
  
  -- 事件记录表
  CREATE TABLE event_record (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    title text,
    event_date timestamptz,
    text text NOT NULL,
    person_id uuid REFERENCES person_profile(id) ON DELETE SET NULL,
    metadata jsonb DEFAULT '{}',
    embedding_id text,
    sensitivity smallint DEFAULT 0,
    deleted smallint DEFAULT 0,
    created_time timestamptz DEFAULT now(),
    updated_time timestamptz DEFAULT now()
  );
  
  -- 分析结果表
  CREATE TABLE analysis_result (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    scenario_text text NOT NULL,
    context_overrides jsonb,
    matched_profiles jsonb DEFAULT '[]',
    retrieved_event_ids jsonb DEFAULT '[]',
    prompt_summary text,
    json_output jsonb,
    text_output text,
    model_meta jsonb,
    confidence_score double precision,
    repair_attempts int DEFAULT 0,
    used_cloud smallint DEFAULT 0,
    status text DEFAULT 'processing',
    deleted smallint DEFAULT 0,
    created_time timestamptz DEFAULT now(),
    updated_time timestamptz DEFAULT now()
  );
  
  -- 审计日志表
  CREATE TABLE audit_log (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    action_type text NOT NULL,
    entity_type text,
    entity_id uuid,
    payload_hash text,
    meta jsonb,
    created_time timestamptz DEFAULT now()
  );
  
  -- 系统配置表
  CREATE TABLE system_preferences (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    key text UNIQUE,
    value jsonb,
    updated_time timestamptz DEFAULT now()
  );
  ```
  
  
  
  ### 索引策略
  
  - `person_profile`：`deleted`，`name` 的 gin 索引（支持全文检索）
  - `event_record`：`deleted`，`person_id`，`event_date`，以及 `embedding` 的向量索引（若使用 pgvector）
  - `analysis_result`：`status` + `created_time` 复合索引
  - `audit_log`：`created_time` 索引
  
  ## 5.5 ORM 实体类（SQLAlchemy）
  
  后端使用 SQLAlchemy 2.0 声明式映射，示例（完整代码见附录）：
  
  python
  
  ```
  # app/db/models/person.py
  from sqlalchemy import Column, Text, SmallInteger, TIMESTAMP, func, Index
  from sqlalchemy.dialects.postgresql import UUID, JSONB, ARRAY
  import uuid
  from app.db.base import Base
  
  class PersonProfile(Base):
      __tablename__ = "person_profile"
      id = Column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
      name = Column(Text, nullable=False)
      aliases = Column(JSONB, default=list)
      profile_json = Column(JSONB, default=dict)
      tags = Column(ARRAY(Text), default=list)
      sensitivity = Column(SmallInteger, default=0)
      deleted = Column(SmallInteger, default=0)
      created_time = Column(TIMESTAMP(timezone=True), server_default=func.now())
      updated_time = Column(TIMESTAMP(timezone=True), server_default=func.now(), onupdate=func.now())
  ```
  
  
  
  其他实体类类似，注意 `event_record` 中向量字段需使用 `pgvector.sqlalchemy.Vector`。

  # 六、LLM 部署架构与混合策略（本地优先，云备选）
  
  ## 本地（On-Prem）—— 推荐用于 V1 的核心策略
  
  - 使用 Ollama 作为本地模型管理与 API 层：在 WSL2 + Docker 下跑 Ollama 容器，加载 `qwen2.5:7b-instruct`、`qwen3:8b`、`chatglm-6b`（或其 GGUF 格式）。
  - 利用显卡（RTX 5070 Ti 16GB）对 7B–8B 模型做混合精度或 INT8/INT4 量化，配合高效推理库（如 vLLM、AWQ 等）降低显存占用。
  - 优点：数据留在本地、成本可控、适合敏感数据场景。
  
  ## 混合/云（按需）
  
  - 对于试验性功能或高并发场景，后端 Orchestrator 可动态路由到云端模型（OpenAI、阿里云 ModelStudio、云上 Qwen 等），并在结果中标注来源与成本（token）。
  - 建议：在 V1 将云调用限定为“非敏感测试”或“滞后校验”，并把成本计入监控。

  # 七、非功能需求（NFR）与运维注意事项
  
  - **性能**：LLM 推理为慢路径（百毫秒至数秒不等），需在前端展示明确的「处理中」提示，并在后端设置合理超时（V1 建议 60–120s）。
  - **可观测性**：记录 request-id、各 LLM 的 model name、prompt hash、response summary、耗时、错误码。提供 Grafana/Prometheus 指标（请求量、平均耗时、LLM 错误率）。
  - **安全**：对敏感文本进行静默脱敏（PII masking）；数据库启用全盘/字段加密；服务间通信 TLS。
  - **扩展性**：采用微服务或模块化结构（PDIS Core 可拆为：orchestrator、profile-service、llm-proxy、vector-sync 等）。
  - **备份与恢复**：Postgres 定期备份，Milvus 元数据与向量备份策略（快照 + 增量）。

  # 十、风险点与缓解措施（面向产品/架构决策者）
  
  1. **模型输出不稳定（hallucination）**
     - 缓解：对关键事实做数据驱动校验（例如数值型证据需与用户上传的数据对齐），并通过“置信度 + 人工复核”流程降低风险。
  2. **成本不可控（云调用）**
     - 缓解：在混合方案中把云调用限制为非关键路径或仅在开发/验证阶段；监控 token 消耗与设置告警。
  3. **本地硬件限制导致并发瓶颈**
     - 缓解：V1 限制并发数、引入请求排队/优先级、并在高并发场景用云端作为“弹性缓冲”。
  4. **数据隐私合规**
     - 缓解：V1 采用本地优先策略并把云端调用限定为脱敏后的非关键数据；后续引入隐私策略与访问控制模块。
  5. **数据一致性与版本管理**
     - 缓解：使用数据库事务保证关键写入一致性；人物侧写版本化通过新记录而非更新实现；审计日志只写不删。
