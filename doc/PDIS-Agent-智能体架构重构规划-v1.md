## 一、为什么要重构？现状 vs Agent架构

### 1.1 当前系统架构问题

| 维度 | 当前流水线架构 | 问题 |
|------|---------------|------|
| 控制方式 | 固定流程 (A→B→C→D) | 无法动态适应复杂场景 |
| 交互模式 | 单轮输入→输出 | 无法持续对话、渐进式澄清 |
| 记忆能力 | 无状态/单次分析 | 人物档案无法持续积累更新 |
| 主动性 | 完全被动响应 | 不会主动追问、提醒、建议 |
| 扩展性 | 增加模块需改流程 | 新增能力需改动整个流水线 |

### 1.2 Agent架构的核心优势

```
传统程序: 你把路修好，程序按路走
Agent系统: 你给目的地，它自己找路
```

**Agent本质特征**:
1. **目标驱动** - 只给目标，自主决策路径
2. **感知-推理-行动循环** - 闭环系统
3. **长期记忆** - 积累人物、历史、偏好
4. **工具使用** - 调用外部能力完成任务
5. **多轮对话** - 渐进式澄清与补充

## 三、核心模块详细设计

### 3.1 感知层 (Perception)

**职责**: 理解用户输入，提取结构化信息

**与旧版区别**:
- 旧版：LLM-A只负责拆分，不做理解
- Agent版：感知层理解"用户意图"，区分不同类型交互

**功能**:
```python
class PerceptionLayer:
    """
    识别用户输入类型：
    - DECISION_REQUEST: 请求决策分析
    - PROFILE_UPDATE: 更新人物档案  
    - FOLLOWUP_ANSWER: 回答追问
    - CHAT: 闲聊/询问
    - CORRECTION: 纠正之前的信息
    """
    def understand_intent(self, user_input: str) -> Intent:
        return {
            "type": "DECISION_REQUEST",  # 或 PROFILE_UPDATE, CHAT等
            "entities": {
                "persons": [...],        # 识别的人物
                "time": [...],           # 时间信息
                "location": [...],       # 地点信息
                "decision": [...],       # 决策核心内容
            },
            "urgency": 0.8,              # 紧急程度
            "sentiment": "concerned",    # 情绪状态
            "needs_clarification": False # 是否需要澄清
        }
```

**存储方案选择** (基于你文档中的经验):

```
第一阶段(MVP): SQLite + JSON文件
- 原因: 数据量小(<10万条)，无需维护额外服务
- 优势: 简单、可调试、与现有Python生态兼容

第二阶段(产品级): PostgreSQL + pgvector
- 原因: 需要语义检索，但数据量中等
- 优势: 关系数据和向量检索统一管理

第三阶段(大规模): 专用向量数据库(Milvus/Qdrant)
- 原因: 数据规模>100万，需要高性能ANN检索
```

### 3.3 推理决策层 (Reasoning/Agent Core)

**核心: Agent决策循环 (ReAct模式)**

```python
class PDISAgent:
    """
    PDIS个人决策小秘书Agent核心
    """
    
    def run(self, user_input: str, session_id: str) -> AgentResponse:
        # 1. 感知理解
        intent = self.perception.understand(user_input)
        
        # 2. 加载相关记忆
        memory_context = self.memory.retrieve(
            query=user_input,
            entities=intent.entities,
            k=5  # 取最相关的5条记忆
        )
        
        # 3. Agent决策循环
        done = False
        iteration = 0
        max_iterations = 5
        
        while not done and iteration < max_iterations:
            iteration += 1
            
            # 3.1 推理：下一步该做什么？
            thought = self.llm.think(
                user_input=user_input,
                intent=intent,
                memory=memory_context,
                previous_actions=action_history,
                available_tools=self.tools.get_descriptions()
            )
            
            # 3.2 决策：选择行动
            action = self.decide_action(thought)
            
            # 3.3 执行行动
            if action.type == "ASK_USER":
                # 需要更多信息，暂停循环等待用户回复
                return AgentResponse(
                    type="CLARIFICATION_NEEDED",
                    question=action.question,
                    session_state=self.get_state()
                )
                
            elif action.type == "ANALYZE_DECISION":
                # 条件充足，进行决策分析
                result = self.analyze_decision(intent, memory_context)
                done = True
                
            elif action.type == "UPDATE_PROFILE":
                # 更新人物档案
                self.memory.update_profile(action.profile_data)
                done = True
                
            elif action.type == "SEARCH_KNOWLEDGE":
                # 检索知识库补充信息
                knowledge = self.tools.search(action.query)
                memory_context.extend(knowledge)
                
            elif action.type == "GENERATE_PLAN":
                # 生成执行计划
                plan = self.generate_execution_plan(intent, memory_context)
                done = True
        
        # 4. 返回结果
        return AgentResponse(
            type="DECISION_COMPLETE",
            content=result,
            confidence=result.confidence
        )
```

**与旧版核心区别**:

| 特性 | 旧版流水线 | Agent版 |
|------|-----------|---------|
| 流程控制 | 固定A→B→C→D | 动态决策循环 |
| 中断处理 | 模拟补充(非真实交互) | 真实多轮对话 |
| 条件判断 | 单点判断(LLM-B) | 每轮循环重新评估 |
| 工具使用 | 无 | 可扩展工具集 |
| 记忆更新 | 分析结束即丢弃 | 持续积累更新 |

## 四、对话流程示例

### 4.1 场景：用户想与老板谈加薪

**第一轮对话**:
```
用户: 我下周二想找老板谈加薪，她最近对我不太满意。

Agent内心活动:
  - 感知: 识别意图=DECISION_REQUEST, 人物=老板, 时间=下周二, 情绪=担忧
  - 记忆检索: 查询"老板"档案 → 未找到
  - 推理: 缺少关键信息(老板性格、不满意原因、加薪依据)
  - 决策: 需要询问用户补充信息

Agent回复:
  我理解你想下周二和老板谈加薪。为了给你更准确的建议，我需要了解一些信息：
  
  1. 你老板的性格大概是怎样的？(比如：强势型、温和型、理性型？)
  2. 她最近对你不满意的具体原因是什么？
  3. 你准备以什么理由来支撑加薪请求？(业绩/市场行情/工作年限？)
  
  你可以先回答其中几个，我会边听边分析。
```

**第二轮对话**:
```
用户: 老板是女的，比较强势，对结果很看重。不满意是因为我上周项目延期了。

Agent内心活动:
  - 感知: 补充了人物性格(强势、结果导向)、事件(项目延期)
  - 记忆更新: 创建"老板"档案，记录性格和事件
  - 推理: 仍缺少信息(加薪理由、延期原因、关系亲疏)
  - 决策: 继续询问，但可以给初步建议

Agent回复:
  了解了，你老板是结果导向型的强势风格，而且对项目延期比较在意。
  
  ⚠️ 基于这个背景，下周二直接谈加薪风险较高。项目延期后立刻谈加薪，
  容易被视为"没有反思问题"。
  
  建议调整策略：
  1. 先准备延期的复盘和改进计划
  2. 以"听取反馈+展示改进"为主要目的
  3. 加薪可以暂缓1-2周，等这段负面印象淡化
  
  不过，如果你有加薪的硬性理由(比如已拿到offer、入职满年限)，
  策略会不同。你有这类理由吗？
```

**第三轮对话**:
```
用户: 我确实入职满两年了，而且最近拿到了一个外部offer，薪资比现在高30%。

Agent内心活动:
  - 感知: 关键信息补充(满两年、有offer、涨幅30%)
  - 记忆更新: 更新用户档案(入职时长)、记录外部offer信息
  - 推理: 条件充足，可以进行完整决策分析
  - 决策: 执行决策分析工具

Agent回复:
  这改变了整体策略。你有外部offer作为谈判筹码，这是很强的底牌。
  
  【决策分析】
  
  可行性: 中高
  信心度: 75%
  
  关键判断依据:
  1. 两年周期是合理的调薪节点
  2. 30%涨幅的外部offer提供了客观市场参照
  3. 但项目延期的负面印象需要妥善处理
  
  执行建议:
  1. 开场先主动提及延期，展示反思态度
  2. 将谈话定位为"职业发展规划讨论"而非"单纯要加薪"
  3. 委婉提及外部机会，表达"优先考虑内部发展"的意愿
  4. 给老板选择空间：匹配offer或制定调薪时间表
  
  风险提示:
  - 如果老板认为你在"威胁"，可能加速你离职
  - 建议准备Plan B：如果谈判失败，是否接受外部offer？
  
  需要我帮你准备具体的沟通话术吗？
```

## 六、关键设计决策

### 6.1 是否使用LangChain/LangGraph？

**建议**: 第一阶段不用框架，第二阶段按需引入

**理由**:
1. 你的文档中提到"先做一个最简单的闭环"
2. 直接使用LLM调用更透明、可调试
3. 框架会增加复杂度，前期阻碍理解
4. 等核心循环稳定后，再考虑框架优化

```
MVP阶段: 直接调Ollama API
产品阶段: 可引入LangGraph做流程编排
```

### 6.2 记忆存储方案？

**建议**: 渐进式演进

```
MVP阶段(现在): SQLite + JSON文件
- 人物档案: SQLite表
- 对话历史: JSON文件
- 检索: 关键词匹配

V2阶段: PostgreSQL + pgvector
- 人物档案: PG表
- 语义检索: pgvector扩展
- 关系型数据和向量统一

V3阶段: 专用向量库(按需)
- 只有当数据量>100万才考虑Milvus
```

### 6.3 模型选择？

**建议**: 延续当前方案，逐步升级

```
当前: Qwen2.5-7B (抽取/轻量任务)
      Qwen3-8B (决策分析)

未来可考虑:
- 感知层: 保持7B模型(结构化抽取不需要大模型)
- 推理层: 可尝试14B模型(更强推理能力)
- 决策层: 可引入GPT-4/Claude(关键决策时调用云端API)
```

## 七、遗留问题与待解决事项

### 7.1 需要进一步明确的决策

1. **云端/本地混合策略**

4. **多用户/社交**
   - V1是否严格单用户？
   - 是否支持"模拟对方视角"？(比如让Agent扮演老板)

## 九、总结

**本次重构的核心转变**:

```
从: "用户输入 → 系统处理 → 输出结果"
到: "用户与Agent持续对话，Agent自主学习、记忆、决策"
```

**关键成功因素**:
1. 保持最小可用，不要一次性引入过多概念
2. 先跑通核心循环，再逐步添加能力
3. 记忆层是核心竞争力，要持续投入
4. 工具化旧代码，不要完全重写

**一句话概括新PDIS-Agent**:
> 一个能记住你的人际关系、理解决策情境、持续对话辅助你做决定的个人AI小秘书。
