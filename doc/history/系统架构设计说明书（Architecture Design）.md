## 1. 架构目标与设计原则

### 1.1 架构目标（**不是分析人，而是服务决策**）

本系统的目标不是构建“人物画像系统”，而是：

> **为你个人的现实行为与沟通提供更稳妥的决策建议**

系统要解决的是真实问题，例如：

- 我现在**要不要**做这件事？
- 如果要做，**什么时候、怎么说、用什么方式**？
- 对方**最可能的反应路径是什么**？
- 如果出现不利反应，我**该如何应对**？

人物性格、行为模式、历史事件：

- **只是决策所依赖的事实素材**
- 永远不作为最终结论直接呈现

------

### 1.2 设计原则（工程视角）

- **决策优先原则**
   数据、模型、侧写都必须服务于“当前决策问题”
- **分层解耦原则**
   决策编排 ≠ LLM 推理 ≠ 存储 ≠ UI
- **本地优先原则**
   默认所有数据、模型、推理都在本地完成
- **可追溯原则**
   每一次决策建议，必须能追溯到：
  - 使用了哪些历史事实
  - 使用了哪个模型
  - 生成时的上下文是什么
- **可进化但不过度设计**
   MVP 阶段只做“你真正会用”的能力

## 3. 分层技术架构说明（与总体图严格一致）

> 本章 **一层一层对照上面的总体图**
>  不新增、不变形

------

### 3.1 客户端 / 使用入口层

**技术选型**

- Web：**Vue 3.x**
- CLI：Python（Click / Typer）
- API 调试：curl / Postman

**选择理由**

- Vue 3：组合式 API，结构清晰，适合长期演进
- 初期 UI 简单，甚至可只用 CLI

**职责边界**

- 只负责输入和展示
- 不做任何决策逻辑

------

### 3.2 应用接入与控制层（Application Layer）

**技术选型**

- **FastAPI 0.110+**
- **Pydantic v2**

**选择理由**

- FastAPI：
  - 类型驱动
  - Swagger 自动生成
  - 非常适合个人/小团队
- Pydantic v2：
  - 性能更好
  - JSON 校验能力强

**职责**

- 接收决策请求
- 参数校验
- 调用决策编排层
- 返回结果

------

### 3.3 决策编排层（Decision Orchestrator）

**技术选型**

- Python Service Layer（纯代码，不是 LLM）

**为什么不用 LLM**

- 决策流程必须**可控、可调试**
- 意图建模如果交给 LLM，会导致行为不可预测

**核心能力**

- 决策类型识别（是否 / 沟通 / 风险）
- 决策状态机
- 调用顺序控制（先查记忆，再推理）

------

### 3.4 认知记忆与检索层（Memory & RAG）

**技术选型**

- **PostgreSQL 15+**
- **pgvector 0.6+**（或 Qdrant 1.8+）

**存什么**

- 人
- 事件
- 决策
- 结果反馈

**为什么这里是“记忆”而不是“画像”**

- 所有数据都是事实或结果
- 画像只是从事实中计算出来的视角

------

### 3.5 推理与模拟层（LLM Engine）

**技术选型**

- **Ollama 3.x**
- 本地模型（Qwen / LLaMA 系列）

**选择理由**

- 本地推理 = 隐私 + 可控
- Ollama 模型管理简单
- 可替换性强

**输出格式**

- **严格 JSON**
- 必须通过 JSON Schema

------

### 3.6 持久化与审计层

#### 为什么用 JSON，而不是 YAML？

**结论：**
 👉 **存储层统一使用 JSON / JSONB，不使用 YAML**

**理由：**

| 对比项       | JSON                | YAML     |
| ------------ | ------------------- | -------- |
| 数据库存储   | ✅ 原生支持（JSONB） | ❌ 不友好 |
| Schema 校验  | ✅ 成熟              | ❌ 较弱   |
| 程序解析     | ✅ 稳定              | ⚠️ 易歧义 |
| 可读性       | 中等                | 较好     |
| 适合长期系统 | ✅                   | ❌        |

**YAML 的正确用途**

- 配置文件
- Prompt 模板
- 非结构化说明

👉 **结论：**

- **数据库：JSON**
- **配置 / Prompt：YAML**

这是工程上的明确分工。

## 4. MVP 范围（技术角度）

**第一阶段只实现：**

- 决策请求 API
- 本地 LLM 推理
- JSON 决策结果
- 本地存储

**不做：**

- 多用户
- 权限体系
- 复杂前端
- 高可用

>  **私人行为决策支持系统**
