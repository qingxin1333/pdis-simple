# 2) 「人物侧写 Prompt 模板」——一次填事实，输出结构化报告（JSON + 文字版）

下面是能让任何 LLM（或你自己的 Agent）**按固定格式自动生成侧写报告**的 Prompt 模板。模板包含：输入字段、输出要求、约束与格式、示例调用。把下面保存为你的 prompt 模板（或直接粘到模型交互框里），每次只填事实就行。

------

## 人物侧写 Prompt 模板（用于生成结构化侧写报告）

**Prompt 使用说明（简短）**

- 填写“事实输入”部分（事实越具体越好：时间、事件、直接观察、证据来源）。
- 模型应输出两部分：`1) JSON 结构化画像`（与上面 JSON 结构一致）和 `2) 人类可读的文字版侧写`。
- 要求模型对“事实”与“推断”明确分离，并对每项推断给出置信度估计（0.0–1.0）。
- 输出语言：中文（简体），JSON 字段须使用英文键名以便系统存储。

------

### —— 模板开始 ——

系统指令（system role，给 LLM）：

> 你是一个精确、客观的人物侧写生成器。输入是“事实集”（含时间、场景、直接观察、引用的文档或对话片段）。输出两部分：**A) 结构化 JSON 画像**（字段应包含 `profile_id, basic_info, main_responsibilities, observable_facts, personality_analysis, thinking_style, historical_behavior, behavior_predictions, monitoring_signals, metadata`）和 **B) 人类可读文字版报告**（章节化：基本信息、事实、性格、思维方式、历史轨迹、风险、结论）。
>  对于每一个**推断**（非直接事实），务必标注置信度（0.0-1.0）并说明支持该推断的证据或逻辑链。不要给执行建议或行动指南（除非输入中特别要求）。保持客观、书面化、适合长期存档。输出 JSON 必须可被直接 parse（无额外注释）。

用户输入（你在执行时替换下列占位内容）：

```
FACTS:
- name: <姓名, 如 侯文杰>
- birth_year: <出生年, 如 1975>
- role: <职位/身份, 如 创始人/董事长>
- company_context: <公司类型/主营, 如 物联网及节能软硬件>
- observed_events: [
    { "date": "YYYY-MM-DD", "context": "会议/场合描述", "observation": "具体事实描述（尽量第一人称或引用原话）", "source": "who/what observed" },
    ...
  ]
- HR_and_financial_patterns: (optional) [
    { "type": "hiring" | "payments" | "reimbursements", "evidence": "描述和证据指向" },
    ...
  ]
- other_relevant_data: (optional free text / attachments references)
```

输出要求（对模型）：

1. 先输出**JSON**（严格可解析）；JSON 字段应至少包含：
   - `profile_id` (string, 格式建议 `person_<surname><given>_<birthyear>_vYYYYMM`)
   - `basic_info` (object: name, gender?, birth_year, role, company_type)
   - `main_responsibilities` (array of strings)
   - `observable_facts` (array of objects: {date, context, fact_statement, source})
   - `personality_analysis` (object: core_traits [array], personality_tags [array], reference_models [array], for each inferred trait include confidence 0.0-1.0 and supporting_evidence)
   - `thinking_style` (array of short strings)
   - `historical_behavior` (array of patterns with confidence and evidence)
   - `behavior_predictions` (object: predicted_actions with likelihood 0.0-1.0 and rationale)
   - `monitoring_signals` (array of strings)
   - `metadata` (source_evidence_summary, generated_at)
2. 在 JSON 之后，输出**人类可读文字版**（中文），章节与之前一致，事实与推断分开，推断后括注置信度与主要证据点。
3. 文风：客观、克制、适合管理层/研究库归档，长度控制在 600–1400 字（文字版）。
4. 输出不要包含执行性建议或情绪化措辞，除非用户在事实输入中明确要求加入“建议”。
5. 若事实不足以支持高置信度推断，模型应明确标注“证据不足”并给出需要的后续证据类型（例如：财务报表、邮件记录、考勤/招聘记录等）。

> **是否需要训练/微调？**——短期**不需要**。优先做 RAG + prompt engineering；当你积累了大量高质量标注数据且有合规保证时，再考虑 LoRA 微调（节约成本但需谨慎）。

# 六、数据流（详细）—— 典型用例：录入事件 → 生成侧写报告

1. 前端填写事件（含 sensitivity flag） -> POST /events
2. 后端将事实写入 Postgres；异步任务：生成 `embedding(text)`；写入向量库，metadata 包含 event_id & sensitivity flag
3. 人工点击“生成侧写”或自动触发 -> Backend 调用 AI Gateway:
   - Gateway 做检索（向量库 top-k，加入结构化事实）
   - build prompt（system + persona + retrieved_docs + schema）
   - 根据 sensitivity 路由到 local LLM or cloud LLM
