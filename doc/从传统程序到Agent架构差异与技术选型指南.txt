# 从传统程序到 Agent：架构差异与技术选型指南

在当前 AI 应用快速发展的背景下，“Agent（智能体）”正在从概念走向工程实践。很多开发者在入门时，往往会混淆几个关键问题：

* 普通程序和 Agent 程序到底有什么本质区别？
* 一个完整的 Agent 系统由哪些核心模块构成？
* 为什么技术选型中更常看到向量数据库和 React，而不是 PostgreSQL 和 Vue？

这篇文章将从工程视角系统梳理这些问题，并给出实际可落地的技术判断。

---

# 一、普通程序 vs Agent 程序

## 1. 控制方式：流程驱动 vs 目标驱动

**普通程序：**

* 基于明确的流程控制（if/else、状态机）
* 输入 → 处理 → 输出
* 每一步逻辑由开发者预先定义

**Agent 程序：**

* 基于目标驱动（Goal-driven）
* 不预设完整路径，而是动态规划
* 能根据环境变化调整行为

👉 本质区别：

* 普通程序：你定义“怎么做”
* Agent：你定义“做什么”，它决定“怎么做”

---

## 2. 执行模式：一次性 vs 循环

普通程序：

```
request → response → 结束
```

Agent 程序：

```
感知 → 推理 → 行动 → 反馈 → 再感知（循环）
```

这是典型的“闭环系统”。

---

## 3. 不确定性处理能力

| 维度 | 普通程序 | Agent  |
| -- | ---- | ------ |
| 输出 | 确定性  | 概率性    |
| 输入 | 完整   | 可不完整   |
| 行为 | 固定   | 可调整    |
| 容错 | 低    | 高（可试错） |

---

## 4. 核心差异总结

| 维度         | 普通程序 | Agent |
| ---------- | ---- | ----- |
| 驱动方式       | 流程   | 目标    |
| 控制逻辑       | 显式编码 | 推理生成  |
| 灵活性        | 低    | 高     |
| 是否具备“决策能力” | 否    | 是     |

---

# 二、Agent 系统的核心组成

一个工程上完整的 Agent，通常包含以下模块：

---

## 1. 目标（Goal）

定义任务最终目的，例如：

* 自动生成报告
* 完成代码修复
* 预订机票

---

## 2. 感知（Perception）

获取外部信息的能力：

* 用户输入
* API 返回
* 文件 / 数据库
* 网页内容

---

## 3. 记忆（Memory）

### 短期记忆

* 当前上下文（context window）

### 长期记忆

* 向量数据库（RAG）
* 历史任务记录

---

## 4. 推理与决策（Reasoning）

Agent 的“核心大脑”，负责：

* 任务拆解（Task Decomposition）
* 规划（Planning）
* 工具选择（Tool Selection）

常见模式：

* ReAct（Reason + Act）
* Plan-and-Execute

---

## 5. 工具调用（Tools）

Agent 的执行能力来源：

* API 调用
* 数据库查询
* Python / Shell 执行
* 浏览器操作

---

## 6. 执行器（Executor）

负责：

* 实际调用工具
* 处理返回结果
* 传回推理模块

---

## 7. 反馈机制（Feedback Loop）

用于：

* 判断任务是否完成
* 是否需要重试或修正

---

## 8. 简化结构图

```
        Goal
         ↓
   Reasoning
         ↓
   Tool Selection
         ↓
     Executor
         ↓
   Environment
         ↓
     Feedback
         ↺
```

# 四、为什么很少提 PostgreSQL？

## 1. PostgreSQL 并不是不能用

通过 pgvector 扩展可以支持：

* 向量存储
* 相似度搜索

---

## 2. 但它不是“最优解”

### （1）定位不同

* PostgreSQL：通用关系数据库
* 向量数据库：高维检索专用系统

---

### （2）性能差异（规模上来之后）

| 场景  | PostgreSQL | 向量数据库 |
| --- | ---------- | ----- |
| 小规模 | ✅          | ✅     |
| 百万级 | ⚠️         | ✅     |
| 亿级  | ❌          | ✅     |

---

### （3）功能差异

向量数据库通常支持：

* ANN 索引（HNSW / IVF）
* 混合检索（向量 + 过滤）
* 分布式扩展

PostgreSQL：

* 可以实现，但需要大量工程工作

---

## 3. 什么时候应该用 PostgreSQL？

适合：

* 数据规模小（<100万）
* 已有 PG 技术栈
* 快速 MVP

---

## 4. 什么时候必须用向量数据库？

适合：

* AI 产品核心依赖检索
* 数据规模大
* 需要高性能与扩展性

---

# 五、为什么前端更常用 React 而不是 Vue？

---

## 1. Vue 并没有过时

Vue 依然具备：

* 高性能
* 易上手
* 在企业系统中广泛使用

👉 问题不是“能不能用”，而是“生态倾向”。

---

## 2. React 在 AI 场景的优势

### （1）AI 生态优先支持

* 大量 SDK 默认 React
* 官方示例基本都是 React

---

### （2）Streaming UI 支持更成熟

Agent UI 核心能力：
👉 流式输出（streaming）

React 提供：

* Suspense
* Server Components
* 流式渲染

---

### （3）组件生态更丰富

例如：

* Chat UI
* Markdown 渲染
* 代码高亮
* Agent 可视化

---

### （4）生态驱动

AI 基础设施公司普遍使用 React，导致：

👉 最佳实践 ≈ React 方案

---

## 3. Vue 适用场景

更适合：

* 后台管理系统
* 中小团队
* 已有 Vue 技术栈
* 不依赖复杂 streaming UI

---

# 六、技术选型的本质逻辑

---

## 1. 向量数据库选择逻辑

```
如果“语义检索是核心能力”
→ 使用专用向量数据库

如果“只是辅助功能”
→ PostgreSQL 足够
```

---

## 2. 前端框架选择逻辑

```
如果“要吃 AI 生态红利”
→ React / Next.js

如果“工程稳定优先”
→ Vue 完全可行
```

---

# 七、推荐技术组合（实战建议）

---

## 1. MVP / 小项目

* PostgreSQL + pgvector
* Vue 或 React
* 单体服务（FastAPI / Node）

---

## 2. AI 产品级系统

* 向量数据库（Pinecone / Weaviate）
* Agent 框架（LangGraph）
* LLM（GPT / Claude）
* 前端（React + Next.js）

---

# 八、总结

Agent 并不是简单的“加了 AI 的程序”，而是范式的变化：

* 普通程序：**流程驱动**
* Agent 系统：**目标驱动 + 自主决策**

而你看到的技术选型差异（PG vs 向量库、Vue vs React），本质上不是“优劣之争”，而是：

👉 **谁更适合当前 AI 工程范式**

---

如果你正在设计自己的 Agent 系统，关键不是“选最流行的”，而是：

> 你的系统，是否真的需要 Agent？
> 你的规模，是否真的需要复杂架构？

这两个问题，决定了 80% 的技术选型。
