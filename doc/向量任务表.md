# 3. 数据分层原则（核心设计约束）

## 3.1 唯一真源原则

PostgreSQL 是唯一业务数据真源：

- 人物档案
- 分析结果
- 原始输入
- 历史记录

Qdrant 仅存：

- 向量
- 结构化 payload

Qdrant 不作为主存。

---

## 3.2 向量写入规则

- 向量写入必须异步
- 不阻塞主流程
- 所有向量必须可重建
- 所有向量必须版本化

---

## 4.3 Qdrant point 设计

```
id: "{pg_id}_v{version}"

payload:
{
  "pg_id": "uuid",
  "entity": "person_profile",
  "version": 1,
  "deleted": 0,
  "name": "张明"
}
```

向量维度必须固定。

# 5. 文本转向量设计（Embedding 规范）

## 5.1 模板生成规则

Worker 统一拼接文本：

顺序必须固定：

1. 姓名 + 角色
2. 年龄 + 心理年龄
3. 身高体重
4. 性格摘要
5. 思考方式
6. 典型行为

示例：

```
张明，35岁，技术部负责人。心理年龄28岁。
身高178厘米，体重80公斤。
性格风险规避型。
思考方式谨慎，决策前反复确认。
典型行为：遇到风险时倾向推迟决策。
```

然后调用 embedding 服务生成：

```
[0.0132, -0.8843, 0.4421, ...]
```

------

## 5.2 向量生成流程

1. 读取主存 profile
2. 拼接模板文本
3. 调用 embedding
4. 校验维度
5. 写入 Qdrant
6. 更新主存 vector_status=ready

------

# 6. 向量版本策略

## 6.1 Point ID 规则

```
{pg_id}_v{version}
```

更新档案：

- version +1
- 生成新向量
- 保留旧版本

------

## 6.2 搜索策略

默认过滤：

- entity = person_profile
- deleted = 0

由主存控制最终使用哪个版本。

------

# 7. 向量检索编排层（Orchestrator）

## 7.1 作用

在 LLM B / LLM D 之前：

1. 将 query 转 embedding
2. 在 Qdrant 搜索 top N
3. 回查 PostgreSQL
4. 生成 analysis_result

------

## 7.2 analysis_result 结构

```
{
  "analysis_id": "uuid",
  "query_text": "...",
  "candidates": [
    {
      "pg_id": "uuid",
      "score": 0.92,
      "profile_snapshot": {...}
    }
  ]
}
```

写入 PostgreSQL 以便审计。

------

# 8. 四阶段流程 + 数据库增强版

## 8.1 完整执行流程

```
graph TD

    A[用户输入] --> B[LLM A 输入拆分]

    B --> C{主存是否存在人物档案?}

    C -- 否 --> D[LLM C 生成人物档案]
    D --> E[写入 PostgreSQL person_profiles]
    E --> F[写入 async_tasks]
    F --> G[Vector Worker 处理任务]
    G --> H[生成向量]
    H --> I[写入 Qdrant]
    I --> J[更新 PostgreSQL vector_status=ready]

    C -- 是 --> K[进入条件判断]

    J --> K[LLM B 条件判断]
    K --> L{是否需要语义检索?}

    L -- 是 --> M[Orchestrator: Query Embedding]
    M --> N[Qdrant 搜索 TopN]
    N --> O[PostgreSQL 回查完整档案]
    O --> P[生成 analysis_result 快照]

    L -- 否 --> P

    P --> Q[LLM D 决策分析]
    Q --> R[ReportManager 保存报告]
    R --> S[输出用户友好报告]

```

# 9. 错误与重试策略

## 9.1 Worker 失败处理

- 网络异常 → 指数退避重试
- 维度不匹配 → 直接失败
- 超过最大重试 → 写入 sync_errors

------

## 9.2 搜索异常

- 记录 analysis_result.status=error
- 不影响主存
- 返回可诊断信息

------

# 10. 测试计划（数据库 + 向量增强版）

## 用例 1：新建人物向量生成

断言：

- person_profiles.vector_status = ready
- qdrant_point_id 非空
- 搜索能返回该 point

------

## 用例 2：人物更新

断言：

- 存在 v1 与 v2
- 默认使用最新版本

------

## 用例 3：相似人物检索

断言：

- 搜索 topN 正确
- PostgreSQL 回查成功

------

## 用例 4：向量异常

模拟 Qdrant 异常

断言：

- async_tasks.status=failed
- sync_errors 记录完整

------

## 用例 5：重建流程

触发 rebuild：

- 新 collection 创建
- 向量数量等于主存实体数量

------

# 11. 验收标准

1. 人物写入 60 秒内完成向量化
2. Orchestrator 可返回 top5
3. 所有版本可追溯
4. analysis_result 可复现
5. 所有异常可追踪
