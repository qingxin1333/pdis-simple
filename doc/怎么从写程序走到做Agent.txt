怎么从写程序走到做Agent

这段时间我在折腾 Agent（智能体）这套东西，一开始其实挺简单的想法：

> 不就是把大模型接进来吗？

结果真正做下来，发现完全不是这么回事。

问题不在“接没接模型”，而在：

* 怎么让系统**自己决定下一步做什么**
* 怎么让它**真的把事情做完，而不是只会说**
* 怎么控制复杂度，不把系统搞崩

这篇文章我不打算讲标准答案，而是把我**自己一路怎么理解、怎么选型、踩了哪些坑**讲清楚。

---

# 一、最开始我犯的一个错误：还在用“写流程”的思路

一开始我写 Agent，其实是这么干的：

```text
用户输入 → 调 LLM → 返回结果
```

然后我开始加一点“智能”：

```text
if 用户要查数据:
    调数据库
elif 用户要分析:
    调 Python
```

写到这里，我就卡住了。

因为你会发现：

👉 **你又在写一个“更复杂的传统程序”**

但 Agent 真正有用的地方，恰恰不是这里。

---

## 后来我才意识到一个关键点

Agent 这套东西，本质不是：

> 写好流程，让程序执行

而是：

> 给一个目标，让系统自己找路径

这个转变其实挺难的，因为它跟我们平时写代码的习惯是反的。

---

# 二、我第一个“能跑”的 Agent，其实非常简单

很多教程一上来就是：

* 多 Agent
* 自动规划
* 长期记忆

我一开始也跟着这么搞，结果就是：

👉 **完全不可控，debug 到崩溃**

后来我反而退了一步，只做了一个最小版本：

---

## 就三样东西

### 1. 一个大模型（负责“想”）

不管你用什么，本质就是让它做三件事：

* 理解当前情况
* 决定下一步干嘛
* 生成调用指令

---

### 2. 几个非常克制的工具

我一开始只给了 4 个：

```text
- read_file
- query_db
- run_python
- call_api
```

一开始我还想加很多工具，后来发现：

👉 工具越多，Agent越容易“乱来”

---

### 3. 一个死循环（核心）

最核心其实就是这段逻辑：

```python
while not done:
    thought = llm(...)
    action = parse(thought)
    result = execute(action)
```

这就是典型的 ReAct 思路。

---

## 一个很直观的变化

从：

```text
我写好每一步 → 程序执行
```

变成：

```text
我只提供能力 → 它自己决定怎么用
```

---

# 三、我在“记忆（RAG）”这里踩的坑

说实话，这一块是我一开始被各种教程“带偏”的地方。

---

## 一开始：直接上向量数据库

我当时的配置是：

* 向量数据库
* embedding
* chunking
* 检索 + rerank

看起来很“标准”，但问题很快就来了：

* 系统变复杂了
* 调试成本很高
* 实际效果…没有想象中那么好

---

## 后来我做了一个很“土”的尝试

👉 全部换成：

```text
PostgreSQL + pgvector
```

原因很简单：

* 系统里本来就有 PG
* 不想多维护一套东西

---

## 结果是：

在数据量不大的时候（几十万级）：

👉 **完全够用**

而且还有几个好处：

* 查询和业务数据可以放一起
* 调试简单很多
* 不需要额外运维

---

## 但问题也确实存在

当数据上来之后（接近百万级）：

* 查询开始变慢
* 相似度结果有点“飘”
* 优化空间有限

这个时候我才换成专门的向量数据库。

---

## 所以我现在的判断很简单

```text
如果“语义检索是核心能力”
→ 上专用向量数据库

如果“只是辅助功能”
→ PostgreSQL 完全够
```

很多教程一上来就推荐向量库，我现在觉得：

👉 **大多数场景是过度设计**

---

# 四、前端我为什么最后选了 React（但其实我更熟 Vue）

这个点也挺真实的。

我原本是更习惯用 Vue 的，写起来也更快。

---

## 一开始我确实用的是 Vue

做一些：

* 后台界面
* 简单交互

完全没问题，甚至更顺手。

---

## 但后来我遇到一个问题：流式输出

Agent UI 有个很关键的体验：

👉 **一边生成，一边显示**

也就是 streaming。

---

### 在 Vue 里不是不能做

但你会发现：

* 要自己处理很多状态
* 一些现成方案不多

---

### 换成 React 之后

明显轻松很多：

* 有现成的 AI SDK
* streaming 支持更自然
* Chat UI 组件很多

---

## 所以我最后的选择是：

```text
后台 / 管理系统 → Vue
Agent / 对话界面 → React
```

---

## 一个挺现实的结论

👉 不是 Vue 不行
👉 是 React 在 AI 这块“默认更省事”

---

# 五、真正让我翻车最多的地方：让 Agent“太自由”

一开始我有个执念：

> 既然是 Agent，就让它自己规划一切

结果就是灾难：

* 乱调用工具
* 步骤越来越离谱
* 很难复现问题

---

## 后来我做了几个限制，系统反而稳定了

---

### 1. 不让它“随便决定一切”

改成：

```text
你只能在这几个步骤里选
```

而不是：

```text
你自己想怎么做
```

---

### 2. 工具必须白名单

不能：

* 随便调 API
* 随便写数据

---

### 3. 保留人工干预

我现在很多场景是：

```text
人定大方向
Agent做执行
```

---

### 4. 强制加可观测性

我后来专门加了日志，记录：

* 每一步它在“想什么”
* 调了什么工具
* 返回了什么

不然真的没法 debug。

---

# 六、如果让我重来一次，我会这么搭

不搞花的，直接说我现在会怎么做。

---

## MVP（最小可用）

```text
后端：
- Python（FastAPI）
- PostgreSQL + pgvector

Agent：
- 直接调 LLM（不一定用框架）

前端：
- 随便（Vue / React）
```

👉 目标：先跑通

---

## 稍微复杂一点

```text
- LLM（GPT / Claude）
- 简单 Agent 框架（比如 LangGraph）
- 工具系统（API / Python）
- 向量库（按需）
- React（如果做对话 UI）
```

👉 目标：可控 + 可扩展

---

# 七、最后一个我觉得最重要的理解

我现在回头看，其实所有问题都可以归结为一句话：

---

## 传统程序

```text
你把路修好
程序按路走
```

---

## Agent 系统

```text
你给一个目的地
它自己找路
```

---

而你做的所有技术选型：

* PostgreSQL 还是向量数据库
* Vue 还是 React

本质上都在回答一个问题：

> ❓ 你愿意让这个系统“自由”到什么程度？

---

# 一个建议（如果你准备动手）

别一上来搞复杂系统。

先做一个最简单的：

```text
读取一个文件 → 总结 → 输出
```

只要你能把这个闭环跑通，而且过程是“它自己决定步骤”：

👉 你已经比大多数教程更接近真正的 Agent 了。
