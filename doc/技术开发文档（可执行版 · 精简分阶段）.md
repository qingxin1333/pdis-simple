# 1. 概述（Scope）

**目标**：构建一个小型、可审计的「人物 + 事件 → 决策建议」系统（PDIS），满足个人决策支持场景：记录人物与事件、基于历史事实做推理、输出结构化 JSON + 人类可读报告、并保证敏感数据本地优先。

**MVP 功能最小集**：

- 人物档案 CRUD（基本信息 + 敏感标记）
- 事件录入（时间、上下文、证据、sensitivity flag）
- 向量化（embedding）并保存到向量库
- AI Gateway：检索相关事实→构建 Prompt→调用 LLM（本地优先）→严格 JSON 输出并保存
- 前端基本 UI（表单录入 + 生成结果查看 + JSON 下载）
- 审计日志（每次生成均记录 input/context/model）

------

# 2. 非功能性需求（NFR）

- **隐私**：敏感数据默认本地存储，不上云（包括 embeddings 除非明确允许）。
- **可追溯**：每次生成必须保存：输入事实 id、模型 id/版本、Prompt 模板、时间戳、输出 JSON 与文本。
- **可扩展**：后续可替换模型、增加向量库副本或云端 fallback。
- **可靠性**：PoC 可单机部署；生产部署需 DB 备份与向量库备份方案。
- **性能**：单用户并发低（MVP 可接受 1-5 RPS）；模型推理为主要延迟点，使用异步任务队列处理耗时请求（Redis + RQ/Celery）。

# 3. 总体架构（文本图）

```
[Browser / CLI] <--HTTPS--> [FastAPI app (API + AI Gateway)]
                                 |
            ------------------------------------------------
            |                      |                      |
        [Postgres]            [Vector DB]           [File Storage/MinIO]
        (facts, profiles)      (Qdrant/Milvus)       (evidence files)
                                 |
                              embeddings
                                 |
                         [Local LLM (Ollama)]
                                 |
                         (optional) Cloud LLM (OpenAI)
```

**说明**：

- 前端（Vue 3）仅呈现/录入；决策逻辑在后端（FastAPI）。
- 向量库保存 embedding + metadata（event_id, sensitivity）。
- AI Gateway 决定路由（sensitivity:true → 本地 LLM only；否则先本地，质量低回退云端）。

# 4. 核心模块详述

## 4.1 前端（Vue 3 + Element Plus）

**职责**：人物/事件录入、生成请求触发、展示 JSON/文字报告、下载与审计查看。
 **实现要点**：

- 事件表单： `title, date, context, observation, source, evidenceFile(upload), sensitivity(boolean)`
- JSON 编辑器页面（Monaco）用于查看/修正 AI 输出。
- 权限：单用户模式初期可免复杂 RBAC。

## 4.2 后端 API（FastAPI）

**职责**：校验请求（Pydantic）、持久化事实、生成 embedding（异步）、AI Gateway 调用、输出验证与保存、审计记录。
 **重要接口（示例）**：

- `POST /api/v1/profiles` — 新建人物档案
- `POST /api/v1/events` — 新建事件（响应立即返回 event_id，异步完成 embedding）
- `POST /api/v1/analyze` — 触发分析（body: person_id/event_ids/context/requester_info） -> 返回 analysis_id，并异步或同步返回结果
- `GET /api/v1/analysis/{id}` — 获取分析结果（JSON + text）

## 4.3 向量化服务（Embedding）

- **模型**：本地 sentence-transformers（all-MiniLM 或更大），或 Ollama 提供 embedding。如果数据敏感，仅本地 embedding。
- **存储**：Milvus 或 Qdrant；metadata 包含 `entity_type,event_id,person_id,created_at,sensitivity`.

## 4.4 AI Gateway

**职责**：

1. 根据 `analysis` 请求检索向量库 top-k（可过滤时间/敏感级别）。
2. 聚合结构化事实（来自 Postgres）与检索文档。
3. 构建 Prompt（system + persona + retrieved docs + schema说明）。
4. 调用本地 LLM（Ollama / local LLaMA），若输出未通过 JSON Schema 校验且策略允许则 fallback 云 LLM（OpenAI）。
5. 用 Pydantic JSON Schema 强校验输出；若不合格，发“修复 prompt”重试（最多 N 次），或标注需要人工介入。
6. 保存输出并写审计日志。

**输出约束**：必须为 JSON（按你定义 schema），并且同时保存人类可读文本。

# 5. 数据模型（核心表 & Pydantic）

## 5.1 最小 SQL 表（Postgres） — 概览

```
-- person_profile
CREATE TABLE person_profile (
  id SERIAL PRIMARY KEY,
  name TEXT NOT NULL,
  birth_year INT,
  role TEXT,
  metadata JSONB,
  sensitivity BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP DEFAULT now()
);

-- event
CREATE TABLE event (
  id SERIAL PRIMARY KEY,
  person_id INT REFERENCES person_profile(id) NULL,
  title TEXT,
  event_date DATE,
  context TEXT,
  observation TEXT,
  source TEXT,
  evidence_paths JSONB,
  sensitivity BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP DEFAULT now()
);

-- analysis_result
CREATE TABLE analysis_result (
  id SERIAL PRIMARY KEY,
  requestor TEXT,
  person_id INT,
  event_ids INT[],
  model_name TEXT,
  model_version TEXT,
  json_output JSONB,
  text_output TEXT,
  status TEXT, -- pending|done|failed
  created_at TIMESTAMP DEFAULT now()
);

-- audit_log
CREATE TABLE audit_log (
  id SERIAL PRIMARY KEY,
  action TEXT,
  actor TEXT,
  entity TEXT,
  entity_id INT,
  payload JSONB,
  created_at TIMESTAMP DEFAULT now()
);
```

## 5.2 Pydantic（示例）

```
from pydantic import BaseModel
from typing import List, Optional
from datetime import date

class PersonCreate(BaseModel):
    name: str
    birth_year: Optional[int]
    role: Optional[str]
    sensitivity: Optional[bool] = False

class EventCreate(BaseModel):
    person_id: Optional[int]
    title: str
    event_date: Optional[date]
    context: str
    observation: str
    source: Optional[str]
    evidence_paths: Optional[List[str]] = []
    sensitivity: Optional[bool] = False

class AnalyzeRequest(BaseModel):
    person_id: Optional[int]
    event_ids: Optional[List[int]]
    question: str
    requestor: Optional[str]
    prefer_local: Optional[bool] = True
```

# 6. 核心流程（序列说明）

**用例：用户想判断是否向老板提议某计划**

1. 前端提交 `AnalyzeRequest`（包含 person_id、相关 event_ids、目标描述/问题文本）。
2. 后端写入 `analysis_result`（status=pending），enqueue 分析任务（RQ/Celery）。
3. Worker：
   - 从向量库检索 top-k 文档（基于 person_id + question）；
   - 拉取关联的结构化事实（Postgres）；
   - 组装 Prompt（system: method/约束；context: facts+retrieved；schema: JSON schema）；
   - 调用本地模型（Ollama）；获得 raw_output；
   - 用 JSON Schema（Pydantic）校验 raw_output；若不合格，触发 repair prompt（最多 2 次）；若仍失败，标注 `status=failed` 并通知人工干预；
   - 若成功，将结果写入 `analysis_result`（status=done），并保存审计日志（audit_log）。
4. 前端轮询或通过通知查看结果并下载 JSON/YAML。

# 7. 部署（PoC 与 生产建议）

## 7.1 PoC （单机，快速上手）

- 使用 Docker Compose 拉起：
  - Postgres（带 pgvector）
  - Qdrant（或 Chroma）
  - FastAPI（uvicorn）
  - Redis（任务队列）
- Local LLM：Ollama 在 Windows 主机运行（并通过 host.docker.internal 访问），或在 WSL 内运行模型容器（若 GPU 可用）。

**docker-compose 片段（关键服务）：**

```
version: '3.8'
services:
  postgres:
    image: ankane/pgvector:latest
    environment:
      - POSTGRES_PASSWORD=changeme
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"

  qdrant:
    image: qdrant/qdrant:latest
    ports:
      - "6333:6333"
    volumes:
      - qdrant_data:/qdrant/storage

  redis:
    image: redis:7
    ports:
      - "6379:6379"

  api:
    build: ./api
    depends_on:
      - postgres
      - qdrant
      - redis
    ports:
      - "8000:8000"

volumes:
  pgdata:
  qdrant_data:
```

## 7.2 生产（建议）

- 数据库：专用 Postgres 主备 +备份策略；pgvector extension。
- 向量库：Milvus/Qdrant 集群（依据数据量）。
- 模型：若本地 LLM 需常用，建议在有 GPU 的服务器上用容器化部署并做 autoscaling（K8s）。
- secrets：HashiCorp Vault 或 equivalent。
- 监控：Prometheus + Grafana；日志：EFK/ELK。

# 8. 安全、隐私与合规要点（必须执行）

1. **敏感标注与策略**：事件/人物带 `sensitivity`，AI Gateway 按标注阻止上云。
2. **最小化上云**：仅当 `sensitivity==false` 且用户明确允许，cloud fallback 才启用。
3. **加密**：DB 字段敏感内容使用 pgcrypto（或字段级加密）；传输 TLS。
4. **审计**：所有分析请求、模型调用、结果导出写入 `audit_log` 并长期保存。
5. **删除策略**：提供“删除人格/事件/全部数据”流程（软删除 + 30 日后永久删除）。

# 9. 测试与验收标准

## 9.1 功能测试

- 人物/事件 CRUD 正常（单元测试覆盖率 > 70%）。
- embeddings 正确生成并写入向量库（验证检索结果包括刚存的 event）。
- `POST /analyze` 能返回 JSON 且能通过 Pydantic 校验。

## 9.2 模型输出质量

- 制定 10 个典型用例（含边界情况）作为回归集合；每次模型或 prompt 变更需跑回归。
- 若模型输出的关键字段缺失率 > 5%，需触发人工审查策略。

## 9.3 安全测试

- 验证敏感数据不被发送到云（断网 + 本地调用检查）。
- 审计日志不可篡改（写入后只追加）。
